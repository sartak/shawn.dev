title: Render static site friction log
date: 2022-04-23

Hi Render!

I recently went through setting up [my blog](https://shawn-dev.onrender.com) as a static site on Render. First and foremost, I'm very impressed with what y'all have put together!

I did want to share my notes on some friction I hit while setting up. Please take all of it with a grain of salt, because none of this diminishes my excitement of your product, or my appreciation for the hard work you've put into your infrastructure and UX. I'm certainly going to migrate my blog from a Linode VM to Render!

If you have any questions, don't hesitate to reach out to me at <a href="mailto:render@shawn.dev">render@shawn.dev</a>.

1. It took me 18 deploys (1 hour) to get my static site generated correctly. The primary reason is that I am old school, so my blog is still built using custom Perl scripts. (I know, I know.) It looks like Render has support for JavaScript, Python, Ruby, Rust, Go, etc., but for Perl I had to bootstrap my own toolchain. Providing a configured [`cpan`](https://perldoc.perl.org/cpan) and/or [`cpanm`](https://metacpan.org/pod/App::cpanminus) out of the box would be a small lift that goes a long way towards plausibly ticking Perl support off your list.
1. Adding a way to inspect the file system after a deploy would be really helpful. I can understand that full `ssh` access would be tricky (especially for the free tier), but a lot of tools log to disk rather than `STDOUT`. So, to be able to inspect some essential output, I had to temporarily add a `cat /path/to/logs/*.log` in my build script (literally with the wildcard because the timestamp was unpredictable), then kick off a new deploy.
1. I reckon the "Using Python version: 3.7.10" deploy logline implies you're intuiting what tech stack I'm using (cool!). But its inaccuracy could erode a bit of confidence. The GitHub page for my repo correctly lists the languages used: "JavaScript: 93.9%, Perl: 3.9%, CSS 1.7%, HTML 0.5%". Given that, I would have expected Render to instead guess JS which would be defensible, or ideally, nothing at all.
1. Every time I deploy, the first line of output is "It looks like we don't have access to your repo, but we'll try to clone it anyway." It then succeeds without any further trouble. It's possible this message shows because I'd never set up a `post-receive` webhook in GitHub; instead I've been doing manual deploys. But as with the previous point, humans (rightly!) have very high expectations for computers being correct. Maybe instead optimistically assume it'll work, and then tell me if and only if it fails? I don't think there's much value in hedging here.
1. When I was specifying the URL for my repository, my first two attempts failed. `git@github.com:sartak/shawn.dev.git` failed silently with zero user feedback. Then I tried `https://github.com/sartak/shawn.dev.git`, which failed with an error that "Render cannot access repo https://github.com/sartak/shawn.dev.git". I had to guess that the problem was the page allows _only_ the `https://github.com/sartak/shawn.dev` form. Seems like an easy win to automatically canonicalize those two URL formats (especially because they're the URLs that GitHub gives you to use for `git clone`).

1. The "Build Command" setting for static sites lists one of its jobs as "installs libraries". Unfortunately I didn't find a clean way to do that. I couldn't use `apt` since the `root` account seems to be unavailable (`sudo` isn't installed and `su` requires an unknown password). I gave up on that, then looked for how I could instead build using a `Dockerfile`. The first place I checked was the [Static Sites](https://render.com/docs/static-sites) docs, specifically under the "Get Started" section. But there's only guidance for popular toolkits rather than bespoke setups. I next checked the [Docker](https://render.com/docs/docker) docs, but I had to read through the whole document before I realized the audience is web services and the like, rather than static sites. Overall I struggled the most here, until I resigned myself to "installing" "libraries" using `curl` 😅 So, it'd be great to give the user some guidance on `apt` specifically, or package management in general. And even better, allow the user to provide a container image instead of a build script. Perhaps the interface could be that the build VM mounts a `/publish` volume that the container copies output files into.

1. Each time I change a setting and it automatically kicks off a deploy, that does spark joy for me. But then it redirects you and requires a few clicks to get to that deploy. If you're going to redirect at all, I'd say sending the user directly to the deploy would be better. (Better still would be to not automatically redirect, but instead pop up an in-page notification with a link to navigate to the deploy). Anyway, the overview page that you get redirected to does have a "Deploy in progress" status badge. I kept trying to click the badge because it has the exact same visual appearance as the dashboard's buttons. So, it'd be great to make the deploy badge link to the deploy, to save the user from that one tiny moment of frustration. (To that end, the clickable areas for the `service-list-container` rows are a little unpredictable)
   <img src="https://shawn-dev.onrender.com/render-friction-log/deploy-in-progress.png" />
1. Early in my setup, I had done two deploys. The first deploy succeeded (because it happened to not do anything yet). The second failed. On the list of deploys, I would have expected a single "rollback to this deploy" button, situated next to that first, successful deploy. Instead what I saw (shown below) was confusing. The first (successful) deploy has no rollback button, while the latest (failed) deploy oddly _does_ have a rollback button. It wasn't until later that I realized that rollback button was disabled. Further testing seems to indicate that perhaps the penultimate deploy _occasionally_ doesn't have a rollback button. If so that is unfortunate because that's almost certainly the only version you'd ever want to rollback to.

All told, one hour to stand up a bespoke app on a new platform is really not bad at all!

I hope this friction log gives you some helpful perspective from a first-time user who hasn't built a mental model for Render. Each one of these points was a small distraction&mdash;the fewer of those in the way between the user and their goal, the better. Driving it down to zero is how you get that feeling of genuine magic. This is the honestly the closest thing I've ever seen to that ideal. 😄

Some other time, I'd be more than happy to jam on UX polish! Such as `incremental` and `decorations` configuration for the deploy log's `xterm-addon-search`, using "skeletons" ([really](https://www.webtips.dev/how-to-create-skeleton-loaders-in-css)) rather than loading spinners throughout the dashboard to improve perceived loading time, fixing the missing arrows pointing to each button from the menu it opened (but then submenus _do_ have the arrow?), more helpful ways to show form validation errors than "There are some errors above", etc.

Thanks for reading! ✌️
